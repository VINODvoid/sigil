## What Judges Look For

### From past hackathon analysis:

**1. Real problem, real users**
Not "what if Solana had X." Evidence of pain, evidence users exist.

**2. Completeness**
Shipped MVP on mainnet/devnet. Not a prototype with fake data.

**3. Business viability**
Required submission content:
- User acquisition strategy
- Monetization plan
- Team credentials

**4. Novel primitives > derivative ideas**
Cypherpunk winners invented new mechanisms. Clones lose.

**5. Impact / ecosystem contribution**
Does this benefit the broader Solana ecosystem?

### Pattern From Grand Champions

- **Unruggable** (Cypherpunk) — hardware wallet primitive
- **TapeDrive** (Breakout) — storage primitive
- **Reflect** (Radar) — stablecoin mechanism primitive

All three: infrastructure primitives that other projects build on top of. **Sigil fits this pattern.**

## What Kills Submissions

**Read these — don't repeat them:**

1. **Demo with mocked data** — judges see through this instantly
2. **Cloning an existing protocol** — "Uniswap but on Solana" loses
3. **No monetization answer** — VCs fund businesses, not features
4. **No user validation** — "I built it because I thought it was cool"
5. **Missing pitch deck** — judges read before watching demo
6. **Late submission** — the platform gets hammered on the last day

---

## What Wins

1. **Start with a primitive thesis** — "this is the missing piece for X"
2. **Ship a real SDK on npm** — developers can install and use during judging
3. **Get 1-2 integration testimonials** — "Team X is using Sigil already"
4. **Clean demo video** — 3 minutes, one flow, show it working live
5. **Pitch in investor language** — TAM, GTM, unit economics, moat
6. **Weekly updates posted** — signals consistency
